# Operating a Freight Platform at 5,000 Requests per Second

## The Load Profile

An aggregate figure such as 5,000 requests per second describes a freight platform only crudely, because that number is the sum of at least three workloads whose shapes have almost nothing in common. Capacity planning that begins from the average will under-provision the peak and over-provision the trough at the same time, so the first task in any scaling playbook is to decompose the aggregate. Search traffic is spiky and human-driven: dispatchers refresh load boards, and a batch of newly posted freight on a busy corridor produces a burst of expensive multi-predicate queries against inventory that is changing underneath them. Telemetry is the opposite. It is machine-generated, write-heavy, composed of small payloads arriving at a near-constant rate, and its only diurnal shape comes from vehicles being on the road rather than from anyone pressing a button. Booking sits between the two: low in volume, sharply peaked during business hours across the operating region, and by far the most costly to get wrong, because a failed search is retried while a failed booking is a lost commercial transaction. The engineering consequence is that these three must be sized, scaled, and degraded independently. A single autoscaling policy driven by platform-wide request rate will starve whichever workload has the tightest latency budget, and a single latency dashboard averaged across all three will report health that no individual user experiences.

## Horizontal Scaling Patterns

Horizontal scaling assumes statelessness, and statelessness is a design constraint rather than an emergent property. Application instances must hold no session affinity, no in-memory work queue whose loss matters, and no locally written file that another instance would need; session and identity state belongs in a shared store so that any instance can serve any request and instances can be terminated without draining special state. In a platform decomposed into roughly 40 microservices, this is what allows the search tier to be scaled out during a board-refresh burst while the settlement tier stays flat. The autoscaling signal matters as much as the mechanism. CPU utilization is a poor proxy for services that spend most of their time waiting on a datastore or a downstream call; concurrency of in-flight requests, queue depth, and consumer lag track saturation far more faithfully for I/O-bound work. Because instances take time to start and warm, scheduled capacity floors ahead of a predictable business-hours peak are usually more effective than reacting to the ramp itself. The real ceiling, though, is almost never the application tier. Each additional replica multiplies connection-pool demand against a shared database, so scale-out can exhaust the datastore's connection limit long before it exhausts compute; a connection proxy, read replicas with an explicit stance on replica lag, and finally partitioning are the escalation path. The trade-off is operational surface: independent scalability means every service carries its own deployment pipeline, dashboards, and failure modes.

## Caching Strategy

Caching on a freight platform is best treated as three separate problems with three different correctness contracts, rather than one cache layer with one expiry setting. Reference data — vehicle and body types, geography, currencies, tariff tables, permission sets — changes rarely and is read constantly, so it tolerates long-lived and even in-process caching. Search results are the opposite: a cached result set that still advertises a load someone has already booked is worse than a slower uncached query, because it converts a latency problem into a trust problem. A common design therefore caches the expensive deterministic sub-computations behind a search — geocoding, lane distance, authorization scopes — while leaving the final result set uncached or cached for only a very short window. Latest-known position per vehicle is a third case that is often mistaken for a cache but is really a materialized view maintained by a stream consumer and read by the API, and it must expose its own age so callers can distinguish a fresh position from a stale one. Invalidation is best driven by domain events rather than by guessing time-to-live values: a booking event evicts the affected search entries directly, which is both faster and more precise than waiting for expiry. The trade-off is that event-driven invalidation makes cache correctness depend on the reliability of the event pipeline, so a bounded expiry should still be kept underneath as a backstop against a lost event, along with request coalescing and jittered expiry to prevent a stampede when a hot key does fall away.

## Zero-Downtime Deployments

Zero-downtime deployment is usually attributed to the rollout mechanism, but the mechanism is the smaller half. Rolling updates replace instances gradually and are the cheapest option; blue-green and canary rollouts hold a full second environment or a small traffic slice and buy a faster, cleaner rollback for riskier changes. What actually makes any of them safe is that consecutive versions can run side by side, because during every rollout both are serving live traffic. That requirement propagates into every contract the service exposes. API changes must be additive, readers must tolerate unknown fields, and event schemas must evolve without reordering or repurposing existing ones. Database changes follow the expand-contract sequence: add the new structure, write to both old and new, backfill historical rows in batches, move reads across once the backfill is verified, stop writing the old structure, and only then drop it. Each step is independently deployable and independently reversible, and no step requires a version of the application that cannot coexist with its predecessor. Renaming or retyping a column in place is the canonical way to break this property. Zero-downtime data synchronization during a migration follows the same logic, with two owners kept consistent through published events and a reconciliation pass that detects divergence. The trade-off is honest and often underestimated: expand-contract turns one migration into a sequence of deployments spread over days, and the discipline that fails most often is finishing the contract phase rather than starting the expand phase.

## Resilience

Resilience at this scale is mostly about bounding the blast radius of a dependency that has become slow rather than one that has stopped. Every outbound call needs an explicit timeout, and those timeouts are best derived from a request budget that descends through the call graph, so that a chain of nested calls cannot collectively exceed what the caller is willing to wait. Retries should be attempted only for idempotent operations and should be governed by a retry budget expressed as a fraction of total traffic, with exponential backoff and jitter; without a budget, a retrying client population turns a partial degradation into a full outage by multiplying load precisely when the dependency has least headroom. Circuit breakers must fail visibly. An open breaker should return an explicit error or a response that is clearly labelled as degraded, never an empty result set that a caller cannot distinguish from a genuine absence of matching freight — a silently empty search is indistinguishable from a market with no loads on it. Bulkheading separates the workloads identified in the load profile: telemetry ingestion and booking should not share thread pools, connection pools, or datastore capacity, so that a surge in position updates cannot consume the resources a booking needs to complete. Admission-level load shedding, which rejects excess work early and cheaply, is preferable to letting queues grow until every request times out. The trade-off throughout is utilization, since reserved and partitioned capacity is by definition capacity that sits idle.

## Observability

The golden signals — latency, traffic, errors, and saturation — remain the right starting frame, provided they are reported per workload class rather than platform-wide. An aggregate of 5,000 requests per second will comfortably hide a saturated booking path behind a healthy search path, and latency must be read as a distribution, because a mean is dominated by the cheapest requests while users notice the tail. Event-driven components need queue-oriented signals in addition: consumer lag should be expressed in time as well as in message count, since a lag of a given number of messages means something entirely different on a slow topic than on a busy one. The most valuable objective in a tracking system is usually freshness — the age of the most recently processed position — and the corresponding alert fires on staleness rather than on errors. This distinction is what catches the characteristic failure of streaming and scheduled workloads: a consumer that has stopped consuming emits no errors at all, so an error-rate alert stays green while data silently ages. Distributed tracing across a platform of roughly 40 microservices makes cross-service latency attributable, and tail-based sampling keeps the slow and failing traces that matter while discarding the bulk of successful ones. Logs should be structured and correlated by trace identifier. Alerting is best tied to symptoms that map to service-level objectives, so that a page corresponds to something a customer can feel rather than to an internal metric crossing an arbitrary line.
